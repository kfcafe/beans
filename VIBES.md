- **`src/main.rs:190`** `consistency` — The `--run` agent spawning logic is duplicated between the `Create` and `Create::Next` command handlers. The same pattern also exists in `src/commands/run.rs` and `src/spawner.rs`, meaning template substitution and agent spawning is implemented in 4+ places.

- **`src/bean.rs:155`** `architecture` — The `Bean` struct has 30+ fields, many of which serve distinct domains: core metadata (id, title, status), verification (verify, fail_first, attempts, history), claim management (claimed_by, claimed_at), memory system (bean_type, last_verified, stale_after, attempt_log), dependency tracking (produces, requires), and hooks (on_fail, on_close). This is a god struct that accumulates every feature.
  > *Consider grouping related fields into sub-structs: `VerifyState { verify, fail_first, attempts, max_attempts, history }`, `ClaimState { claimed_by, claimed_at }`, `MemoryState { bean_type, last_verified, stale_after, attempt_log }`. This preserves the flat YAML serialization via `#[serde(flatten)]` while making the code more navigable.*
- **`src/bean.rs:468`** `resource-cleanup` — `Bean::to_file()` uses `std::fs::write()` directly, which is not atomic. If the process crashes mid-write (or an agent is killed during `bn close`), the bean file will be corrupted — left partially written. Since beans are the source of truth and index is derived, this can cause data loss.
  > *Write to a temporary file in the same directory, then `fs::rename()` to the target path. This makes writes atomic on the same filesystem. Bean #84.2 tracks this exact issue.*
- **`src/index.rs:1`** `resource-cleanup` — `Index::save()` also uses non-atomic `fs::write()`. The index is a cache and can be rebuilt, so corruption is recoverable — but it still causes confusing errors when agents crash during index writes, requiring manual `bn sync` to fix.
  > *Use write-to-temp-then-rename pattern for index writes too, or at minimum document that `bn sync` recovers from index corruption.*
- **`src/commands/run.rs:918`** `function-length` — `run_ready_queue_direct()` is ~430 lines managing the entire ready-queue dispatch loop: tracking completed beans, spawning threads, waiting for results, handling errors and keep-going logic. Combined with `run_single_direct()` (~200 lines), the direct-mode execution path is complex and dense.
  > *Extract the thread pool management into a generic `WorkPool` struct that handles spawn/wait/capacity, separate from the bean-specific dispatch logic. The bean-specific result handling could be a callback or separate function.*
- **`src/commands/run.rs:46`** `architecture` — `run.rs` defines its own `BeanAction`, `SizedBean`, `Wave`, `DispatchPlan`, and `AgentResult` types. Meanwhile `src/spawner.rs` defines a near-identical `AgentAction` enum and its own `AgentProcess`/`CompletedAgent` types. The two systems have overlapping responsibilities: `spawner.rs` handles template-mode spawning while `run.rs` handles both template and direct mode, making `Spawner` partially redundant.
  > *Unify `BeanAction` and `AgentAction` into one type. Decide whether `Spawner` or `run.rs` owns the spawning lifecycle — currently `run.rs` bypasses `Spawner` entirely for direct mode, making `Spawner` dead weight for the primary code path.*
- **`src/spawner.rs:265`** `architecture` — `claim_bean()` and `release_bean()` shell out to `bn claim` and `bn claim --release` as subprocesses of themselves — the CLI calling its own binary. This is fragile (depends on `bn` being in PATH, adds process overhead) and creates a circular dependency pattern. It means spawner only works when `bn` is installed, not when running from `cargo run`.
  > *Call `cmd_claim()` and `cmd_release()` directly as library functions instead of shelling out. The functions already exist in `src/commands/claim.rs` and are exported.*
- **`src/commands/run.rs:121`** `test-coverage` — `cmd_run()` — the core agent orchestration entry point — has no integration tests that exercise the direct-mode path (spawning real agents). - **`src/commands/close.rs:540`** `edge-cases` — When `bean.to_file()` and `fs::rename()` (archive) happen in sequence, if the rename fails after to_file succeeds, the bean is marked as closed in its original location but never archived. - **`src/commands/close.rs:282`** `error-handling` — The pre-close hook error handling silently passes on execution failure: `Err(e) => { eprintln!(...); true }`.   > *Make the behavior configurable or at least log at warning level. Consider returning an error when the hook can't be spawned (distinct from the hook returning non-zero).*
- **`src/commands/verify.rs:1`** `test-coverage` — The `verify` command module is only 47 lines with zero tests. It delegates to `close.rs`'s `run_verify()` but adds its own behavior (history recording). This critical path — running verification without closing — has no test coverage.

- **`src/commands/status.rs:1`** `test-coverage` — `cmd_status` — the primary project overview command used to triage work — has no unit tests. This is a user-facing entry point that aggregates claimed/ready/blocked/goal beans.

- **`src/mcp/tools.rs:840`** `edge-cases` — `truncate_str` slices a `&str` at byte offset `s.len() - max`, which can panic if that offset falls in the middle of a multi-byte UTF-8 character. `String::from_utf8_lossy` in the caller can produce multi-byte replacement characters (U+FFFD = 3 bytes), so `s.len()` reflects byte count, not char count. Slicing at an arbitrary byte boundary is undefined behavior in safe Rust via a panic.
  > *Use `s.char_indices()` to find a char-aligned boundary, or use `s.chars().rev().take(max).collect()` to truncate safely at character boundaries.*
- **`src/mcp/tools.rs:488`** `edge-cases` — The `handle_close_bean` function truncates combined stdout+stderr output with `&combined[combined.len() - 2000..]`. Same UTF-8 byte-boundary panic risk as `truncate_str` — if the 2000th-from-last byte falls mid-character, this will panic at runtime.
  > *Use a char-boundary-safe truncation method, e.g., find the nearest char boundary with `combined.floor_char_boundary(combined.len() - 2000)` (nightly) or iterate `char_indices()` to find a safe split point.*
- **`src/commands/close.rs:493`** `edge-cases` — Verify output truncation uses `&stdout[..MAX_OUTPUT_BYTES]` which slices at a raw byte offset (64KB). If the verify command outputs multi-byte UTF-8 (e.g., emoji, CJK), this can panic on a character boundary. The `stdout` variable is a `String` obtained from `from_utf8_lossy`, so it's valid UTF-8 but may have multi-byte chars.
  > *Use `stdout.char_indices().take_while(|(i, _)| *i < MAX_OUTPUT_BYTES).last()` to find the nearest safe boundary, or use `String::truncate` with a char-aligned offset.*
- **`src/commands/plan.rs:190`** `input-validation` — The `bean_path_str` (file path from `find_bean_file`) is interpolated directly into a shell command string via `format!("pi @{} {}", bean_path_str, escaped_prompt)`. While the prompt is shell-escaped, the file path is NOT escaped. If a bean slug or `.beans/` path contains shell metacharacters (spaces, backticks, `$`, etc.), this creates a command injection vector. The slug is derived from user-provided titles and while sanitized, the `beans_dir` path itself could contain unsafe characters.
  > *Apply `shell_escape()` to `bean_path_str` as well: `format!("pi @{} {}", shell_escape(&bean_path_str), escaped_prompt)`.*
- **`src/spawner.rs:152`** `resource-cleanup` — When `Spawner::spawn()` successfully starts a child process but `register_agent` or the HashMap insert fails (e.g., OOM), the child process is leaked — it keeps running with no tracking. The claim was already acquired, so the bean stays claimed with no way to release it. Additionally, the log file handles are moved into the child process but not explicitly tracked for cleanup.
  > *Wrap the post-spawn registration in a cleanup guard pattern: if registration fails, kill the child process and release the bean claim.*
