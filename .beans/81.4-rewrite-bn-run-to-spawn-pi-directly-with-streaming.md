## Task
Rewrite src/commands/run.rs so that `bn run` spawns pi agents directly
(not via shell template) with JSON streaming, timeout monitoring, and
real-time event parsing.

This is the core integration — bn run becomes what deli run was.

## Changes

### 1. Add --json-stream flag to CLI
In src/cli.rs, add to the Run subcommand:
```rust
#[arg(long)]
json_stream: bool,
```

### 2. Add --timeout and --idle-timeout flags
```rust
#[arg(long, default_value = "30")]
timeout: u64,  // minutes

#[arg(long, default_value = "5")]  
idle_timeout: u64,  // minutes
```

### 3. Replace shell template spawning with direct pi spawn
Instead of:
```rust
Command::new("sh").args(["-c", &cmd]).spawn()
```

Do:
```rust
let bean_file = find_bean_file(beans_dir, &bean.id)?;
let context = get_context(&bean.id);
let mut cmd = Command::new("pi");
cmd.args(["--mode", "json", "--print", "--no-session"]);
if !context.is_empty() { cmd.args(["--append-system-prompt", &context]); }
cmd.arg(&format!("@{}", bean_file.display()));
cmd.arg("Implement and bn close");
cmd.stdout(Stdio::piped());
```

### 4. Monitor with timeout and parse events
For each spawned agent:
- Use timeout::monitor_process to read stdout with timeouts
- Parse each line with pi_output::parse_agent_event
- Track cumulative tokens/cost per bean
- Emit stream::StreamEvent for --json-stream mode

### 5. Wave execution with thread pool
Keep the existing wave computation (compute_waves). Replace run_wave to:
- Spawn up to max_jobs agents in parallel threads
- Each thread: claim bean → spawn pi → monitor → check if closed → release on failure
- Emit BeanStart/BeanDone/RoundStart/RoundEnd events

### 6. Keep backward compatibility
If config.run is set (template mode), use the old behavior.
If config.run is NOT set but pi is on PATH, use direct spawn.
This way existing users aren't broken.

## Files
