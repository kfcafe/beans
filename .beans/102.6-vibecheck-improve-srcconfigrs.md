## Task
Improve `src/config.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] consistency — `src/config.rs:202-208`
  - `Config::save()` uses `fs::write()` (non-atomic) while `Bean::to_file()` uses `atomic_write()` (write-to-temp-then-rename). When multiple agents run concurrently, a crash mid-write could corrupt `config.yaml`, which contains `next_id` — a critical monotonically increasing counter. A corrupted config means all subsequent bean creation fails.
  - Suggested fix: Change `Config::save()` to use `crate::util::atomic_write()` for consistency and crash safety.
- [medium] input-validation — `src/config.rs:190-199`
  - `resolve_extends_path` resolves config inheritance paths relative to the project root, but does not validate that the resolved path stays within the project directory. A malicious `extends: ["../../../etc/some-config.yaml"]` in `.beans/config.yaml` could read arbitrary YAML files from anywhere on disk. While `canonicalize()` in `load_with_extends` prevents symlink attacks, it doesn't prevent upward traversal. This is a path traversal risk in shared/cloned repos.
  - Suggested fix: After resolving the extends path, verify it stays within the project root or home directory. Reject paths that traverse above the project root (after canonicalization).
- [low] input-validation — `src/config.rs:100-140`
  - The `extends` config field resolves paths using `~/` expansion and relative paths, but after canonicalization there is no check that the resolved path stays within a safe boundary (e.g., the user's home directory). A malicious `.beans/config.yaml` from a cloned repo could set `extends: ["../../other-project/.beans/config.yaml"]` to inherit `run` templates from outside the project. The `run` template is later executed via `sh -c`.
  - Suggested fix: After canonicalizing extends paths, verify they stay within the user's home directory or the project tree. At minimum, warn when loading extends from outside the project root.
