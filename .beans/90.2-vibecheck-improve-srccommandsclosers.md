## Task
Improve `src/commands/close.rs` based on vibecheck findings.

## Findings (top 20)
- [high] function-length — `src/commands/close.rs:267-680`
  - `cmd_close` is ~413 lines long (excluding tests), handling verify execution, history recording, circuit breaker logic, on_fail processing, worktree merging, archiving, on_close actions, auto-close parent, and index rebuilding — all in a single function. This violates single responsibility and makes the function extremely difficult to follow, test in isolation, or modify safely.
  - Suggested fix: Extract into focused helpers: `handle_verify_failure(bean, verify_result) -> Result<()>`, `handle_verify_success(bean, verify_result) -> Result<()>`, `archive_bean(beans_dir, bean, reason) -> Result<()>`, `process_on_close_actions(bean, project_root)`, `handle_worktree_merge(bean) -> Result<bool>`. The main `cmd_close` should orchestrate these at a high level.
- [high] single-responsibility — `src/commands/close.rs:1-2869`
  - close.rs is 2,869 lines — the largest file in the codebase. It combines the core close command logic (~680 lines), 7 private helper functions, and ~2,100 lines of tests (82 tests). While thorough testing is good, the file's sheer size makes navigation difficult and signals that the production code needs decomposition.
  - Suggested fix: Extract the helper functions (`run_verify`, `all_children_closed`, `auto_close_parent`, `find_root_parent`) into a separate `src/commands/close_helpers.rs` or `src/verification.rs` module. Move test sections to `tests/close_test.rs` for the integration-style tests.
- [low] dead-code — `src/commands/close.rs:41`
  - The `stderr` field in `VerifyResult` is marked `#[allow(dead_code)]` — it's captured but never read. This is a code smell indicating the struct was designed with fields that aren't actually used.
  - Suggested fix: Remove the `stderr` field entirely, or use it in error reporting. Currently `output` (combined stdout+stderr) is used everywhere instead.
- [low] logging — `src/commands/close.rs:330-330`
  - The `BEANS_AGENT` environment variable value is read and stored directly into bean run records (`bean.history`) without sanitization. While this is low risk (it's a local env var), if a malicious agent sets `BEANS_AGENT` to a very long string or string with control characters, it gets persisted to disk in the bean file and could corrupt YAML rendering or cause parsing issues on read.
  - Suggested fix: Validate or truncate the `BEANS_AGENT` value — limit to reasonable length (e.g., 100 chars), strip control characters, and validate it matches an expected pattern (alphanumeric + hyphens).

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
