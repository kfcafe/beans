## Task
Improve `src/commands/run.rs` based on vibecheck findings.

## Findings (top 20)
- [high] function-length — `src/commands/run.rs:1-2216`
  - run.rs is 2216 lines containing the orchestrator (wave computation, ready-queue scheduling, process spawning, JSON stream events, template mode, direct mode) plus ~500 lines of tests. The `run_single_direct` function alone (~200 lines) builds commands, spawns processes, sets up monitoring, parses JSON events, and handles file locks — too many concerns.
  - Suggested fix: Split into modules: `run/plan.rs` (plan_dispatch, compute_waves), `run/spawn.rs` (run_single_direct, template execution), `run/queue.rs` (ready-queue logic). The private `AgentResult` type is also duplicated with `spawner.rs`'s `CompletedAgent`.
- [medium] resource-cleanup — `src/commands/run.rs:1082-1095`
  - In `run_single_direct`, file locks are acquired pre-emptively (lines 1082-1095) but if the pi process fails to spawn (lines 1119-1130), the locks are never released — the cleanup at line 1290 (`release_all_for_bean`) only runs after the monitoring section, which is skipped on early returns.
  - Suggested fix: Use a guard/RAII pattern that releases locks on drop, or restructure so lock release happens in all exit paths. An `if file_locking { release_all_for_bean(...) }` call should be added to each early-return `AgentResult`.
- [medium] test-coverage — `src/commands/run.rs:1082-1290`
  - `run_single_direct` — the core function that spawns pi agents, monitors them with timeouts, parses JSON events, and captures tokens/costs — has zero unit tests. It's only tested indirectly through integration tests that require `pi` to be installed. This is the most critical code path for agent orchestration.
  - Suggested fix: Extract the JSON event parsing callback (lines 1184-1250) into a testable function. Mock the process spawning to test timeout handling, lock acquisition/release, and error paths.
- [low] dead-code — `src/commands/run.rs:1476-1480`
  - `find_bean_file` is a public wrapper that just calls `crate::discovery::find_bean_file`. The doc comment says "Public wrapper for use in tests" but it re-exports functionality already available via `bn::discovery::find_bean_file`. Test code can import the original directly.
  - Suggested fix: Remove the wrapper and have tests import from `crate::discovery` directly.
- [low] consistency — `src/commands/run.rs:830-850`
  - `run_wave_direct` uses `Arc<Mutex<Vec>>` and thread joining for parallel execution, while `run_ready_queue_direct` uses `mpsc::channel` for completion notification. These are two different parallelism patterns for the same conceptual operation (running beans in parallel). The wave-based direct mode is also unreachable when the ready-queue mode is used (which is the default for `SpawnMode::Direct`).
  - Suggested fix: Remove `run_wave_direct` since `run_ready_queue_direct` subsumes its functionality with better granularity. The wave-based approach is only used for `SpawnMode::Template`, which has its own `run_wave_template`.
