## Task
Improve `src/config.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] input-validation — `src/config.rs:98-190`
  - `Config::load_with_extends` processes the `extends` field by resolving paths and loading parent configs. While it tracks `seen` paths to prevent infinite loops, it doesn't limit recursion depth, and the path resolution (supporting `~/` expansion) doesn't validate that resolved paths stay within expected directories. A malicious config could reference arbitrary files on the filesystem.
  - Suggested fix: Add a maximum extends depth (e.g., 5 levels). Validate that resolved extends paths are within the home directory or project directory. Log a warning when loading extends from unexpected locations.
- [medium] input-validation — `src/config.rs:189-198`
  - `rules_path()` resolves the `rules_file` config value with no path traversal protection. A malicious or compromised `.beans/config.yaml` can set `rules_file: "../../../../etc/passwd"` (or any absolute path) and the file will be read and injected into agent context output via `load_rules()` in `commands/context.rs` and `commands/run.rs`. Unlike bean ID validation and ctx_assembler, which both enforce path boundaries, rules_file does not canonicalize or restrict the resolved path to the project directory.
  - Suggested fix: After resolving the path, canonicalize it and verify it starts with the canonicalized beans_dir (or project root). Reject paths containing `..` components or that resolve outside the project boundary. Apply the same pattern used in `assemble_context()` which already does `canonical.starts_with(&canonical_base)`.
- [medium] input-validation — `src/config.rs:163-179`
  - `resolve_extends_path()` resolves config inheritance paths relative to the project root (or `~/`) without any boundary checking. A malicious `.beans/config.yaml` can use `extends: ["../../../etc/shadow"]` to attempt reading arbitrary files on the filesystem. While the file must be valid YAML to be parsed, the read itself succeeds, and error messages may leak path/content information. This contrasts with ctx_assembler which explicitly validates paths stay within the project.
  - Suggested fix: After resolving and canonicalizing the extends path, validate it stays within an expected directory (project root or home directory). Reject paths that resolve outside these boundaries.

- Prefer small, focused changes.
