## Task
Improve `src/hooks.rs` based on vibecheck findings.

## Findings (top 20)
- [high] input-sanitization — `src/hooks.rs:284-315`
  - `expand_template` performs naive string replacement of user-controlled variables (`{title}`, `{output}`, `{children}`, etc.) into shell command templates without any shell escaping. These expanded strings are then passed to `sh -c` in `execute_config_hook`. A bean title like `"; rm -rf / #` would be injected directly into the shell command. The `{output}` variable (verify command output) is also attacker-influenced — a verify command could emit shell metacharacters in its stdout/stderr.
  - Suggested fix: Shell-escape all variable values before substitution. Use the existing `shell-escape` crate dependency: `shell_escape::escape(Cow::from(v))` for each variable. Alternatively, pass variables as environment variables to `sh -c` instead of interpolating them into the command string.
- [medium] input-sanitization — `src/hooks.rs:301-301`
  - The `{output}` truncation at 1000 chars uses byte indexing (`&v[..1000]`) which will panic on multi-byte UTF-8 strings if the 1000th byte falls in the middle of a multi-byte character. This is a denial-of-service vector if verify commands emit non-ASCII output.
  - Suggested fix: Use `v.chars().take(1000).collect::<String>()` or `v.char_indices().nth(1000).map(|(i,_)| &v[..i]).unwrap_or(v)` to truncate at a valid character boundary.
- [medium] input-validation — `src/hooks.rs:346-369`
  - `execute_config_hook` is fire-and-forget — it spawns a child process via `sh -c` but never calls `wait()` on it. This creates zombie processes if the parent keeps running. Over many bean close/fail operations, accumulated zombies could exhaust PID space. The function also has no trust check — unlike `execute_hook` which checks `is_trusted()`, config hooks (on_close, on_fail, post_plan) execute unconditionally since they come from config.yaml.
  - Suggested fix: Either wait for the child in a background thread, or use `Command::new().spawn()` and store the `Child` handle for later reaping. Consider whether config hooks should also respect the trust model — currently a malicious config.yaml can execute arbitrary commands without `bn trust`.

- Prefer small, focused changes.
