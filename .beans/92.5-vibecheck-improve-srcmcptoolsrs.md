## Task
Improve `src/mcp/tools.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] edge-cases — `src/mcp/tools.rs:840-846`
  - `truncate_str` performs byte-level string slicing (`&s[s.len() - max..]`) without checking UTF-8 character boundaries. If the slice point falls inside a multi-byte UTF-8 character (e.g., emoji or CJK text in verify output), this will panic at runtime. The same issue exists at line 488 in `handle_close_bean` with `&combined[combined.len() - 2000..]`. The close.rs `truncate_to_char_boundary` helper already solves this correctly but isn't reused here.
  - Suggested fix: Replace byte-offset slicing with a char-boundary-safe truncation. Reuse the existing `truncate_to_char_boundary` pattern from `close.rs`, or use `s.char_indices().rev().nth(max)` to find a safe cut point. Apply to both `truncate_str` (line 842) and the inline truncation in `handle_close_bean` (line 488).
- [medium] input-validation — `src/mcp/tools.rs:469-510`
  - The `handle_close_bean` function in the MCP server duplicates the close logic from `commands/close.rs` but skips several safety checks: no pre-close/post-close hooks, no circuit breaker, no on_fail actions, no worktree handling, and no structured history recording. An MCP client calling `close_bean` bypasses these protections, potentially closing beans that should have been rejected by hooks or hitting the circuit breaker.
  - Suggested fix: Refactor `handle_close_bean` to call the shared `cmd_close` function from `commands/close.rs` instead of reimplementing close logic. Capture stdout/stderr from the shared function rather than running verify independently. This ensures all security and lifecycle checks apply uniformly regardless of entry point.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
