## Task
Improve `src/mcp/tools.rs` based on vibecheck findings.

## Findings (top 20)
- [high] input-validation — `src/mcp/tools.rs:840-846`
  - `truncate_str` slices a `&str` at a raw byte offset (`s[s.len() - max..]`) without checking UTF-8 character boundaries. If the cut point falls inside a multi-byte UTF-8 character (e.g., emoji, CJK, accented chars), this panics at runtime. The same codebase already has a correct implementation — `truncate_to_char_boundary` in `close.rs` — but `mcp/tools.rs` doesn't use it. This is reachable via the MCP `verify_bean` tool when verify command output contains non-ASCII characters.
  - Suggested fix: Replace the raw byte-offset slicing with a char-boundary-safe helper (either reuse `truncate_to_char_boundary` from `close.rs`, or use `s.floor_char_boundary(max)` on nightly/equivalent logic). E.g.: `let start = s.len().saturating_sub(max); let start = start + s[start..].chars().next().map_or(0, |c| c.len_utf8()); format!("...{}", &s[start..])`.
- [medium] input-validation — `src/mcp/tools.rs:488-536`
  - The MCP `close_bean` tool handler executes verify commands via `sh -c` from bean files without additional authorization. Any MCP client (IDE, Cursor, Claude Desktop) that can call `create_bean` followed by `close_bean` can achieve arbitrary command execution. While verify commands are a core design feature, the MCP server provides no additional authentication, rate limiting, or command sandboxing beyond the stdio transport layer. A compromised or malicious MCP client gets full shell access with the user's privileges.
  - Suggested fix: Consider adding an allowlist of permitted verify commands in MCP mode, requiring explicit user confirmation before running verify via MCP (similar to the hook trust model), or documenting this as an accepted risk with guidance to only connect trusted MCP clients. At minimum, log all verify command executions from MCP for auditability.
