## Task
Improve `src/bean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] architecture — `src/bean.rs:151-296`
  - The `Bean` struct has 35+ fields spanning task metadata, verification state, claim lifecycle, memory system, and token tracking. It's a god struct — every feature adds more fields, making serialization/deserialization fragile and the struct hard to reason about. Adding a field requires updating `try_new`, `apply_value`, tests, `IndexEntry::from`, and the serde skip annotations.
  - Suggested fix: Group related fields into sub-structs: `VerifyState { verify, fail_first, attempts, max_attempts, history, outputs }`, `ClaimState { claimed_by, claimed_at }`, `MemoryState { bean_type, last_verified, stale_after, paths, attempt_log }`. Use `#[serde(flatten)]` to maintain YAML compatibility.
- [medium] edge-cases — `src/bean.rs:487-499`
  - `Bean::hash()` comment says "transient fields cleared" for optimistic locking, but the implementation serializes the full bean including `updated_at`, `claimed_at`, and `history` — all of which change on every operation. Two beans identical in content but with different timestamps will produce different hashes, making optimistic locking overly aggressive.
  - Suggested fix: Create a canonical form that excludes volatile fields (`updated_at`, `claimed_at`, `history`, `attempts`, `tokens_updated`) before hashing. Or hash only the fields that represent user intent (title, description, verify, dependencies, etc.).
- [low] comprehensibility — `src/bean.rs:90-103`
  - The comment on `OnFailAction` says "Declarative action to run when a bean's verify command fails" but the enum is named `OnFailAction` while the section header comment above it says "OnCloseAction". The `OnCloseAction` definition immediately follows at line 105, creating confusion about which is which when scanning the file.
  - Suggested fix: Move the `OnCloseAction` section header comment (line 90: `// OnCloseAction`) to just above the `OnCloseAction` enum (line 105), not above `OnFailAction`.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
