## Task
Improve `src/bean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] error-handling — `src/bean.rs:478-486`
  - `Bean::hash()` uses `serde_json::to_string(&canonical).unwrap()` — a panic in library code. While `Bean` is always serializable in practice, `unwrap()` in a public method used for optimistic locking could crash the process if the bean ever contains non-serializable data (e.g., a future field with `f64::NAN`).
  - Suggested fix: Change return type to `Result<String>` or use `expect("Bean must be JSON-serializable")` with a comment explaining why this is safe. Alternatively, keep `unwrap` but document the invariant.
- [medium] comprehensibility — `src/bean.rs:145-300`
  - The `Bean` struct has 35 fields. While each field is individually documented, the struct as a whole is a god object — it represents task metadata, verification state, claim state, memory system data, agent orchestration config, and output capture all in one type. This makes it hard to understand which fields relate to which subsystem.
  - Suggested fix: Group related fields using marker comments or consider wrapping clusters into sub-structs (e.g., `VerifyState { verify, fail_first, attempts, max_attempts, history }`, `ClaimState { claimed_by, claimed_at }`, `MemoryState { bean_type, last_verified, stale_after, paths, attempt_log }`). The serialization can stay flat via serde's `#[serde(flatten)]`.
- [low] comprehensibility — `src/bean.rs:462-470`
  - `to_file` clones the entire bean just to remove the `description` field for frontmatter serialization. For a 35-field struct with vectors and optional types, this is wasteful. More importantly, the logic is subtle: description is removed from YAML and placed as the markdown body, but only for `.md` files with a description present.
  - Suggested fix: Consider using `#[serde(skip)]` on a wrapper or serializing with a custom serializer that omits description, rather than cloning the full bean. At minimum, add a comment explaining the clone-and-remove pattern.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
