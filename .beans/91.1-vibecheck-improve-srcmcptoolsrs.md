## Task
Improve `src/mcp/tools.rs` based on vibecheck findings.

## Findings (top 20)
- [high] input-sanitization — `src/mcp/tools.rs:840-845`
  - `truncate_str` slices a `&str` at a raw byte offset (`s[s.len() - max..]`) without checking for UTF-8 character boundaries. If the string contains multi-byte UTF-8 characters (e.g., emoji, CJK, accented characters) and the byte offset lands in the middle of a multi-byte sequence, this will panic at runtime. The `close.rs` module has the same pattern but uses a dedicated `truncate_to_char_boundary` helper — this function does not.
  - Suggested fix: Use the `truncate_to_char_boundary` pattern from `close.rs`, or use `s.char_indices()` to find the nearest safe boundary. For example: `let start = s.char_indices().rev().nth(max).map(|(i, _)| i).unwrap_or(0); format!("...{}", &s[start..])`.
- [high] input-sanitization — `src/mcp/tools.rs:486-491`
  - The `handle_close_bean` MCP tool handler has the same UTF-8 slicing bug: `&combined[combined.len() - 2000..]` can panic on multi-byte characters. This code path is reachable by any MCP client sending a `close_bean` tool call when the verify command produces non-ASCII output.
  - Suggested fix: Replace the raw byte slice with a char-boundary-safe truncation. The `truncate_to_char_boundary` helper already exists in `close.rs` — extract it to `util.rs` and reuse it here.
- [high] input-validation — `src/mcp/tools.rs:468-508`
  - The `handle_close_bean` MCP tool handler passes `verify_cmd` directly to `sh -c`. While the verify command comes from the bean file (not the MCP request), the `force` parameter is client-controlled — a compromised MCP client can close any bean without verification by passing `force: true`. The MCP protocol doesn't have authentication, so any connected client has full bean lifecycle control.
  - Suggested fix: Consider adding an MCP-level authorization or at minimum logging when `force: true` is used from MCP. Document the trust model: MCP clients are fully trusted since they run on localhost via stdio. If the MCP server is ever exposed over a network, this becomes critical.
- [medium] input-validation — `src/mcp/tools.rs:337-366`
  - The `create_bean` MCP tool handler does not validate the `title` field beyond checking it exists. An MCP client could submit extremely long titles, titles containing control characters, or titles that produce adversarial slugs. The CLI entry point likely has shell-level length limits, but the MCP server accepts arbitrary JSON input with no length or content validation on `title`, `description`, `verify`, `acceptance`, or `deps`.
  - Suggested fix: Add length limits and basic sanitization for string inputs in MCP tool handlers. At minimum, validate `title.len() <= 500`, reject null bytes and control characters, and validate `deps` entries with `validate_bean_id`.
