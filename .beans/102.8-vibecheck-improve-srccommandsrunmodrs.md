## Task
Improve `src/commands/run/mod.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] consistency — `src/commands/run/mod.rs:63-74`
  - `BeanAction` enum (Implement/Plan) is semantically identical to `AgentAction` in `src/spawner.rs:25-37`. Both have the same variants, same Display impl, same purpose. Having two parallel types for the same concept creates confusion about which to use and requires converting between them.
  - Suggested fix: Consolidate into a single `AgentAction` type (or `BeanAction`) in a shared location like `bean.rs` or a new `types.rs`, then use it in both `spawner.rs` and `commands/run/`.
- [low] dead-code — `src/commands/run/mod.rs:484-486`
  - `pub fn find_bean_file` is a trivial wrapper around `crate::discovery::find_bean_file` that adds no logic, no error context, no transformation. It just delegates. Any callers should import from `discovery` directly.
  - Suggested fix: Remove this wrapper and have callers import `crate::discovery::find_bean_file` directly.
- [low] architecture — `src/commands/run/mod.rs:64-78`
  - `run/mod.rs` defines `BeanAction` enum (Implement/Plan) while `src/spawner.rs` defines a near-identical `AgentAction` enum with the same two variants and the same Display implementation. Both types serve the same purpose — indicating whether an agent should implement or plan a bean — but are separate, requiring conversion between them if they interact.
  - Suggested fix: Unify into a single `AgentAction` type (either in `spawner.rs` or a new shared module like `types.rs`) and use it in both the run command and the spawner.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
