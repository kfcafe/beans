## Task
Improve `src/bean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] error-handling — `src/bean.rs:477-484`
  - `Bean::hash()` calls `serde_json::to_string(&canonical).unwrap()` in non-test code. While `Bean` should always be serializable, `unwrap()` in production code is a panic risk if a future field change introduces a non-serializable type. This is used for optimistic locking — a panic here would crash the CLI.
  - Suggested fix: Return `Result<String>` instead, or use `.expect("Bean must be JSON-serializable")` with a clear panic message at minimum. Better: propagate the error since callers like `from_file_with_hash` already return `Result`.
- [medium] comprehensibility — `src/bean.rs:160-290`
  - The `Bean` struct has 34 fields, making it a "god struct". Fields span at least 5 concerns: identity (id, title, slug), workflow (status, priority, claimed_by, claimed_at), verification (verify, fail_first, attempts, max_attempts, history), tree structure (parent, dependencies, produces, requires), and memory system (bean_type, last_verified, stale_after, paths, attempt_log). This makes it hard to understand what subset of fields matters for any given operation.
  - Suggested fix: While splitting the struct is invasive, improve comprehensibility by: (1) grouping fields with clear section comments (already partially done), (2) adding a module-level doc explaining the field groups and their lifetimes, and (3) considering builder patterns or field-group structs for construction (e.g., `VerifyConfig { verify, fail_first, max_attempts, on_fail }`).
- [low] dead-code — `src/bean.rs:486-493`
  - `Bean::from_file_with_hash()` and `Bean::apply_value()` are public methods that are never called outside of `bean.rs` (only in their own tests). These were likely built for a conflict resolution or API feature that hasn't been connected yet.
  - Suggested fix: Either connect these to their intended callers (the API module's commented-out `mutations` submodule) or mark them `pub(crate)` and add a doc comment noting they're for future use. Dead public API surface creates maintenance burden and false expectations.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
