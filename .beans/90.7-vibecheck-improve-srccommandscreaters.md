## Task
Improve `src/commands/create.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] single-responsibility — `src/commands/create.rs:1-1958`
  - create.rs is 1,958 lines combining bean creation logic, child ID assignment, on-fail parsing, the `create_next` command, and 42 tests. The create flow itself (~300 lines) isn't too long, but the file is bloated by test infrastructure that could be shared.
  - Suggested fix: Move tests to a dedicated test file. Consider extracting `assign_child_id` and `parse_on_fail` into utility modules since they're reusable parsing/ID logic, not create-specific.
- [low] input-validation — `src/commands/create.rs:1-100`
  - Bean creation via both CLI and MCP accepts an arbitrary `verify` string that is later passed to `sh -c`. While this is by design (verify commands must be shell commands), there is no warning, logging, or confirmation when a verify command contains potentially dangerous patterns (e.g., `rm -rf /`, `curl | sh`, writing to system paths). The trust model for hooks exists but there is no equivalent guardrail for verify commands.
  - Suggested fix: Consider adding a lightweight scan for obviously dangerous patterns in verify commands (recursive delete, piped-to-shell patterns, system path writes) and printing a warning. This is informational — the user should still be able to set any command, but a warning helps catch mistakes or malicious bean creation.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
