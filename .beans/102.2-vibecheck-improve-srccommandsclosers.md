## Task
Improve `src/commands/close.rs` based on vibecheck findings.

## Findings (top 20)
- [high] single-responsibility — `src/commands/close.rs:267-720`
  - `cmd_close` is a ~460-line function that handles verify execution, output truncation, failure recording, circuit breaker logic, on_fail processing, worktree merging, archiving, hook execution, on_close actions, config hooks, worktree cleanup, auto-close parent, and index rebuilding. This makes it extremely difficult to understand, test individual behaviors, or modify one concern without risking others.
  - Suggested fix: Extract into focused helper functions or a `CloseWorkflow` struct: `run_and_record_verify()`, `archive_bean()`, `fire_hooks()`, `process_on_fail()`, `check_circuit_breaker()`. The existing `run/` module split is a good pattern to follow — do the same for close.
- [high] function-length — `src/commands/close.rs:268-660`
  - `cmd_close` is approximately 400 lines handling at least 8 distinct responsibilities in a single function: pre-close hook execution, verify command execution, failure recording with history, circuit breaker logic, on_fail action processing, worktree merge, archiving, on_close action processing, and parent auto-close. This makes the function extremely difficult to test in isolation, reason about, or safely modify.
  - Suggested fix: Extract into focused functions: `run_pre_close_hook()`, `run_and_record_verify()`, `check_circuit_breaker()`, `process_on_fail()`, `archive_bean()`, `process_on_close_actions()`, `try_auto_close_parent()`. The main `cmd_close` should become a thin orchestrator calling these steps in sequence.
- [high] single-responsibility — `src/commands/close.rs:1-3330`
  - close.rs is 3,330 lines — the largest file in the project. It mixes core close/verify logic (~400 lines), parent auto-close helpers (~140 lines), utility functions (~60 lines), the `cmd_close_failed` function, and ~2,680 lines of tests. The 5:1 test-to-code ratio is excellent for coverage but having everything in one file makes navigation difficult.
  - Suggested fix: Split into a `close/` module directory: `close/mod.rs` (cmd_close orchestrator), `close/verify.rs` (run_verify, VerifyResult), `close/archive.rs` (archiving, parent auto-close), `close/tests.rs` (all test code). This mirrors the existing `run/` module structure.
- [medium] file-structure — `src/commands/close.rs:1-3330`
  - At 3330 lines, `close.rs` is the largest file in the codebase. Roughly 2500 lines are tests (from line ~800 onward). The test module alone is larger than most entire source files. This makes navigation difficult and increases merge conflict risk since every close-related change touches this one file.
  - Suggested fix: Split into a module directory `commands/close/` with: `mod.rs` (public API), `verify.rs` (verify execution and recording), `archive.rs` (archiving and auto-close-parent), `hooks.rs` (close-specific hook orchestration), and a `tests/` subdirectory. Bean 94 already exists for this.
- [medium] comprehensibility — `src/commands/close.rs:41-47`
  - The `VerifyResult` struct has `#[allow(dead_code)]` on its `stderr` field, meaning stderr is captured but never used independently. The `output` field combines stdout+stderr. This suggests the struct was designed for a future use case that was never implemented, or the design changed and the field was left behind.
  - Suggested fix: Either use `stderr` independently (e.g., for structured error reporting in run history) or remove it and the `#[allow(dead_code)]` annotation. The current state is confusing — it looks like unused code that someone chose to keep "just in case."
- [medium] input-sanitization — `src/commands/close.rs:49-80`
  - The `run_verify` function passes the bean's verify command directly to `sh -c` without any validation or sandboxing. While this is by-design (verify commands are meant to be shell commands), any actor with write access to `.beans/` files can execute arbitrary code via the verify field. This is particularly concerning in shared/cloned repositories where `.beans/` files may come from untrusted sources. The MCP server's `handle_close_bean` and `handle_verify_bean` also execute verify commands from bean files without additional checks.
  - Suggested fix: Document this as a known trust boundary. Consider adding a `bn trust` check before executing verify commands from bean files that were not created locally (e.g., based on git blame or a signed-by field). At minimum, warn users when `.beans/` files exist at `bn init` time in cloned repos.
- [medium] error-handling — `src/commands/close.rs:282-290`
  - Pre-close hook errors (not executable, timeout, I/O failure) are silently treated as "pass" with `Err(e) => { eprintln!(...); true }`. This means if a security-critical pre-close hook can't execute (permissions error, binary missing, timeout), the bean is closed without the hook's validation. A hook that fails to run is semantically different from a hook that approves.
  - Suggested fix: At minimum, distinguish between "hook doesn't exist" (which should pass) and "hook exists but failed to execute" (which should block or warn prominently). Consider a `strict_hooks` config option that treats execution errors as failures for security-sensitive deployments.
- [low] performance — `src/commands/close.rs:121-152`
  - `all_children_closed` calls `Index::build(beans_dir)` (full rebuild from disk) every time it's checked, even though it may be called multiple times during a single `cmd_close` invocation (once per closed bean's parent). Each rebuild reads and parses every bean file on disk.
  - Suggested fix: Pass the already-loaded index as a parameter, or cache the result. The comment says "Always rebuild the index fresh" but the staleness concern could be addressed by rebuilding once at the start of `cmd_close` and reusing it.
- [low] dead-code — `src/commands/close.rs:41`
  - `VerifyResult.stderr` field is annotated with `#[allow(dead_code)]` — it's captured during verify execution but never read anywhere. The `output` field already contains combined stdout+stderr, making `stderr` redundant.
  - Suggested fix: Remove the `stderr` field and its `#[allow(dead_code)]` annotation. If stderr-only access is needed later, it can be extracted from the combined output or re-added.
