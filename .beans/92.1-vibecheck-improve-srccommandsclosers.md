## Task
Improve `src/commands/close.rs` based on vibecheck findings.

## Findings (top 20)
- [high] function-length — `src/commands/close.rs:265-670`
  - `cmd_close` is ~405 lines long and handles verify execution, failure recording, circuit breaker logic, on_fail processing, history recording, output capture, worktree merge, archiving, hook execution, on_close actions, worktree cleanup, and auto-close parent. This violates single responsibility and makes the function extremely difficult to follow, test in isolation, or modify safely.
  - Suggested fix: Extract into focused helper functions: `handle_verify_failure()`, `handle_verify_success()`, `archive_bean()`, `process_on_close_actions()`. The existing `run_verify()` extraction is a good pattern — continue it for the other concerns.
- [high] single-responsibility — `src/commands/close.rs:1-2869`
  - close.rs is 2869 lines — the largest file in the codebase. It contains verify execution, failure formatting, parent auto-close, root parent traversal, the main close command, AND 41 test functions (roughly 2000 lines of tests). The close logic, archiving logic, and test code should be separated.
  - Suggested fix: Extract archiving into a shared `archive` module (it's duplicated with tidy.rs). Move tests to a dedicated `tests/close_test.rs` or `close/tests.rs` submodule. Keep `cmd_close` as a thin orchestrator calling focused helpers.
- [medium] architecture — `src/commands/close.rs:567-590`
  - Bean archiving logic (compute archive path, create directories, rename file, set is_archived, save) is duplicated between `cmd_close` (lines 567-590), `auto_close_parent` (lines 193-218), and `cmd_tidy` (tidy.rs lines 201-247). Three separate implementations of the same 7-step archiving workflow.
  - Suggested fix: Extract into a shared `archive_bean(beans_dir, bean, bean_path)` function in a common module (e.g., `discovery.rs` or a new `archive.rs`). All three callers should use it.
- [medium] edge-cases — `src/commands/close.rs:369`
  - `Config::load(beans_dir).map(|c| c.max_loops).unwrap_or(10)` is called inside the per-bean verify failure path. This reloads and re-parses config.yaml from disk on every single failed verify, even in a batch close. It also silently falls back to 10 if config is missing, which may surprise users who set a different value in extends configs (since `Config::load` doesn't resolve extends).
  - Suggested fix: Load config once at the start of `cmd_close` (using `Config::load_with_extends`) and pass it down. This is both more efficient and correct (respects config inheritance).
- [medium] graceful-degradation — `src/commands/close.rs:49-67`
  - The `run_verify` function executes user-supplied shell commands via `sh -c` with no timeout. A malicious or buggy verify command (e.g., `sleep infinity`) will block the `bn close` process indefinitely. The hooks system correctly implements a 30-second timeout, but verify commands — which run arbitrary user-specified shell code — have no equivalent protection. The MCP `handle_close_bean` and `handle_verify_bean` have the same issue.
  - Suggested fix: Add a configurable timeout (e.g., 300s default) to verify command execution. Use `child.try_wait()` in a loop with a deadline, similar to the pattern in `hooks.rs:execute_hook`. Kill the process if it exceeds the timeout.
- [low] dead-code — `src/commands/close.rs:41`
  - The `stderr` field on `VerifyResult` has `#[allow(dead_code)]` — it's captured but never used. The combined `output` field (stdout+stderr) is used instead for backward compatibility. This is a sign that the struct was designed for future use but the field was never wired in.
  - Suggested fix: Either use `stderr` for separate error reporting (e.g., showing stderr distinctly in failure output) or remove it and the `#[allow(dead_code)]` annotation.
- [info] input-validation — `src/commands/close.rs:472-475`
  - The `on_close` `Run` action executes shell commands from bean YAML (`on_close: [{run: "command"}]`). While this is gated behind the trust system (`is_trusted`), the trust check uses the project root, not the specific bean. A bean file contributed via PR could contain `on_close` actions that execute when any trusted project closes that bean. The trust model is coarse-grained — it's all-or-nothing for the entire project.
  - Suggested fix: Document that `on_close` run commands in bean files are subject to the same trust model as hooks. Consider adding a per-bean trust mechanism or requiring explicit approval for beans with `on_close` run actions. At minimum, log the command being executed to stderr (which is already done).
