## Task
Improve `src/bean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] architecture — `src/bean.rs:155-290`
  - The `Bean` struct has 35+ fields spanning at least 5 distinct domains: core metadata (id, title, status, priority), verification state (verify, fail_first, attempts, max_attempts, history, checkpoint), claim management (claimed_by, claimed_at), memory system (bean_type, last_verified, stale_after, attempt_log, paths), and hooks (on_fail, on_close). This "god struct" accumulates every feature into one flat type, making it hard to understand which fields relate to which subsystem.
  - Suggested fix: Group related fields into sub-structs: `VerifyState { verify, fail_first, attempts, max_attempts, history, checkpoint }`, `ClaimState { claimed_by, claimed_at }`, `MemoryState { bean_type, last_verified, stale_after, attempt_log, paths }`. Use `#[serde(flatten)]` to preserve flat YAML serialization while improving code navigability.
- [low] error-handling — `src/bean.rs:482-491`
  - `Bean::hash()` uses `.unwrap()` on `serde_json::to_string()`. While serialization of a Bean is extremely unlikely to fail (all fields are standard types), this is production code used for optimistic locking. A panic here would crash the entire process rather than propagating an error.
  - Suggested fix: Change to `-> Result<String>` and propagate the error, or document why the unwrap is safe. Since `hash()` is called from conflict resolution paths, a panic there could leave bean files in an inconsistent state.
- [low] edge-cases — `src/bean.rs:443-446`
  - `Bean::from_file` reads the entire file into memory before parsing. There is no file size check, so a malicious bean file with a very large YAML document (gigabytes) could cause out-of-memory. The `ctx_assembler::read_file` correctly limits to 1MB, but bean file loading has no such protection.
  - Suggested fix: Add a file size check (e.g., 10MB limit) before reading bean files. This prevents memory exhaustion from maliciously large `.beans/` files in cloned repos.
- [low] comprehensibility — `src/bean.rs:107-113`
  - The doc comment on `OnFailAction` says "Declarative action to run when a bean's verify command fails" but the section header comment above it (line 104) says "OnCloseAction". This is misleading — `OnFailAction` and `OnCloseAction` are two different enums defined adjacent to each other, and the section header labels the wrong one.
  - Suggested fix: Fix the section header comment from "OnCloseAction" to "OnFailAction", and add a separate section header before the actual `OnCloseAction` enum definition. Add a blank line between the two enum blocks for visual separation.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
