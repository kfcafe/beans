## Task
Improve `src/config.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] edge-cases — `src/config.rs:119-158`
  - Config inheritance via `extends` compares field values against default functions to decide whether to inherit. If a user explicitly sets `max_tokens: 30000` (the default value) in their local config, it will still be overridden by a parent config because the check `config.max_tokens == default_max_tokens()` returns true. This is a semantic error — explicit local values should always win.
  - Suggested fix: Track which fields were explicitly set (e.g., via a `#[serde(default)]` wrapper that records whether the value came from the YAML file or from the default function). Alternatively, use `Option<u32>` for all inheritable fields and only inherit when `None`.
- [low] input-validation — `src/config.rs:135-159`
  - The `resolve_extends_path` function resolves config inheritance paths relative to the project root or home directory with no restrictions on traversal. A `config.yaml` with `extends: ["../../../etc/some-yaml-file"]` could read arbitrary YAML files on the filesystem. While this requires the attacker to control `config.yaml` (which is project-owned), it's a risk in shared/cloned repositories where config files may not be reviewed.
  - Suggested fix: Validate that resolved extends paths remain within the project directory or a well-known config location (e.g., `~/.config/beans/`). Canonicalize the resolved path and check it starts with the project root or home config directory before reading.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
