## Task
Improve `src/commands/run.rs` based on vibecheck findings.

## Findings (top 20)
- [high] function-length — `src/commands/run.rs:178-433`
  - `run_once` is ~255 lines. It interleaves planning, dry-run printing, JSON stream emission, template-mode execution, and direct-mode execution with nearly identical result-handling blocks duplicated between `SpawnMode::Direct` and `SpawnMode::Template`. The two match arms share ~50 lines of near-identical reporting/counting logic.
  - Suggested fix: Extract the shared result-aggregation logic into a helper. The SpawnMode-specific branches should return a unified `Vec<AgentResult>` and a `had_failure` bool, then a single reporting block handles output. This eliminates duplication and cuts the function in half.
- [medium] single-responsibility — `src/commands/run.rs:1-2216`
  - run.rs (2,216 lines) combines dispatch planning, wave computation, two different spawning modes (template + direct), ready-queue scheduling, single-agent execution with process monitoring, context assembly, output formatting, and 31 tests. It's effectively 5-6 modules in one file.
  - Suggested fix: Split into: `run.rs` (orchestration + cmd_run), `dispatch.rs` (plan_dispatch, compute_waves, SizedBean), `spawn_direct.rs` (pi spawning + monitoring), `spawn_template.rs` (legacy template spawning). Keep the types (RunArgs, Wave, DispatchPlan) in run.rs or a shared types file.
- [medium] performance — `src/commands/run.rs:816-887`
  - `run_wave_direct` uses a busy-wait loop with `thread::sleep(200ms)` polling `is_finished()` on thread handles. This is inefficient — it wakes up every 200ms regardless of whether work completed. With many concurrent agents, this adds unnecessary CPU overhead.
  - Suggested fix: Use a channel-based approach (like `run_ready_queue_direct` already does) or `thread::JoinHandle::join()` with a separate monitoring thread. The ready-queue implementation is already better — consider removing the wave-based direct mode entirely and always using ready-queue.
- [medium] resource-cleanup — `src/commands/run.rs:1083-1316`
  - In `run_single_direct`, if the process is killed due to timeout (MonitorResult::TotalTimeout or IdleTimeout), file locks are released but the child process may leave behind orphaned subprocesses. The function calls `child.wait()` after kill, but doesn't send SIGTERM before SIGKILL or wait for process group cleanup.
  - Suggested fix: Send SIGTERM first, wait briefly, then SIGKILL if needed. Consider using process groups (`setsid`) when spawning so the entire process tree can be cleaned up. The timeout module may already handle some of this — verify the kill path.
- [medium] test-coverage — `src/commands/run.rs:1083-1316`
  - `run_single_direct` (233 lines) — the function that actually spawns `pi` agents, monitors their output, handles timeouts, parses JSON events, tracks tokens/cost, and releases locks — has zero direct unit tests. It's only exercised through integration tests that depend on `pi` being installed. This is the highest-risk function in the orchestration layer.
  - Suggested fix: Extract the monitoring/event-parsing logic into a testable function that takes a `BufRead` + mock child process. Test timeout paths, JSON parsing edge cases, and lock cleanup independently of having `pi` on PATH.
- [low] consistency — `src/commands/run.rs:1463-1465`
  - `run.rs` re-exports `find_bean_file` as a public wrapper around `crate::discovery::find_bean_file` with zero added logic. This creates a confusing second path to the same function, used only in tests within this module.
  - Suggested fix: Remove the wrapper. Tests should use `crate::discovery::find_bean_file` directly, or keep it `pub(crate)` / `#[cfg(test)]` to signal it's test-only.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
