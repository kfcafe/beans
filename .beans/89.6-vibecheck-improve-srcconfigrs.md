## Task
Improve `src/config.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] input-validation — `src/config.rs:95-130`
  - The `extends` config mechanism loads and executes config from arbitrary file paths, including paths resolved with `~/`. If a project's `config.yaml` is cloned from an untrusted repository, the `extends` field could point to paths that read sensitive configs or inject malicious `run`/`plan` templates. While cycle detection exists, there is no validation that extended paths stay within reasonable boundaries (e.g., project directory).
  - Suggested fix: Add a warning when `extends` references paths outside the project directory. Consider requiring explicit trust (like hooks) before inheriting from external configs, or at minimum printing the resolved extends paths on first load.
- [low] edge-cases — `src/config.rs:116-148`
  - The config inheritance merge logic checks each field against its default value (e.g., `config.max_tokens == default_max_tokens()`) to decide whether to inherit from the parent. This means if a user explicitly sets `max_tokens: 30000` (the default value) in their local config, it will still be overridden by a parent config's non-default value — the user's explicit choice is indistinguishable from "not set".
  - Suggested fix: Use `Option<T>` for inheritable fields in the raw config, then resolve to concrete values after merge. Or use a separate `ConfigOverrides` struct that tracks which fields were explicitly set.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
