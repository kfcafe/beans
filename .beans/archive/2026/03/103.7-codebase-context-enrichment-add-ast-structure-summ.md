## What

Enrich `bn context <id>` output with a lightweight structural summary of the files a bean references — function signatures, type definitions, and imports. This gives agents better situational awareness without dumping entire files.

## Why

From HN (melvinroest): "I vibe coded my own static analysis program where I just track my own function calls. It outputs a call graph." Agents work better when they understand the structure of what they're touching, not just raw file contents.

## Steps

1. Modify `src/commands/context.rs` to optionally include a structural summary section
2. For each file path referenced in the bean description, extract:
   - Function/method signatures (name + parameters + return type)
   - Type/struct/enum definitions (name + fields/variants)
   - Import statements
3. Add this as a "## File Structure" section before the full file contents
4. Use simple regex-based extraction (not a full AST parser — keep it dependency-free):
   - Rust: `fn name(`, `pub fn name(`, `struct Name`, `enum Name`, `use `
   - TypeScript/JS: `function name(`, `export function`, `class Name`, `interface Name`, `import `
   - Python: `def name(`, `class Name`, `import `, `from ... import`
5. Add `--structure-only` flag to skip full file contents and just show structure
6. Keep it optional — if extraction fails for a file type, just skip the structure section

## Acceptance Criteria
- `bn context <id>` includes a structural summary for recognized file types
- Supports at minimum: Rust (.rs), TypeScript (.ts/.tsx), Python (.py)
- Structure section appears before full file contents
- `--structure-only` flag outputs just the structure, not full files
- Unrecognized file types are silently skipped (no error)
- JSON mode includes structure in output

## Don't
- Don't add tree-sitter or any AST parsing dependency — use regex
- Don't fail if structure extraction doesn't work for a file — graceful fallback
- Don't change the existing file content output format
