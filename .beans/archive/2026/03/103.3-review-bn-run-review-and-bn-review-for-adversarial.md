## What

Add adversarial review as a first-class concept. After an agent closes a bean, a review agent (ideally a different model) checks the implementation against the bean's spec in a fresh context.

Two entry points:
- `bn run --review` — auto-review after every close during a run
- `bn review <id>` — manually trigger review of any bean

## Why

Verify gates check mechanical correctness (does the test pass?) but not semantic correctness (does this actually do what was asked? is the code good?). Review fills that gap. From HN (virgilp): "ask other AI, 'is this implementation conforming to the spec or did it miss things?'"

## Steps

1. Add `--review` flag to `Run` command in `src/cli.rs`
2. Add `Review` command to `Command` enum in `src/cli.rs`:
   - `bn review <id>` — review a bean's implementation
   - `--model <model>` — optionally use a different model
   - `--diff` — include only the git diff since bean was created/claimed
3. Create `src/commands/review.rs`:
   - `cmd_review()` — assembles review context (bean description, git diff, verify output), spawns review agent
   - Review agent verdict: approve / request-changes / flag
   - On request-changes: reopen bean with review notes injected, label `review-failed`
   - On flag: add label `needs-human-review`, bean stays closed
   - On approve: add label `reviewed`, done
4. Add review config to `src/config.rs`:
   ```yaml
   review:
     run: "pi -p 'review bean {id}: ...'"  # review agent command template
     max_reopens: 2  # don't loop forever
   ```
5. Wire `--review` in `src/commands/run/mod.rs`: after a bean closes, if --review is set, call `cmd_review()`
6. Accumulated knowledge integration: when review reopens a bean, inject review notes so the next implementation agent sees "Review failed because: X"

## Acceptance Criteria
- `bn review <id>` spawns a review agent with bean description + git diff as context
- Review verdict (approve/request-changes/flag) updates bean labels
- request-changes reopens the bean with review notes
- `bn run --review` auto-triggers review after each close
- max_reopens prevents infinite review loops
- Review config is optional — everything works without it configured

## Don't
- Don't make review mandatory — it's opt-in via flag or config
- Don't block on review agent — it runs like any other agent
- Don't modify the close command itself — review hooks in after close completes
