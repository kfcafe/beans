## Task
Improve `src/mcp/tools.rs` based on vibecheck findings.

## Findings (top 20)
- [high] architecture — `src/mcp/tools.rs:460-557`
  - `handle_close_bean` reimplements bean close logic (verify, archive, auto-close parent) instead of delegating to `cmd_close` or a shared function. The MCP version skips: run history recording, circuit breaker checks, on_fail processing, hook execution, on_close actions, worktree handling, and fail-first enforcement. This means MCP clients get silently different (less safe) behavior than CLI users.
  - Suggested fix: Extract the core close logic from `cmd_close` into a library function (e.g., in `api/`) that both the CLI command and MCP handler call. The MCP handler should not reimplement verification and archiving.
- [high] architecture — `src/mcp/tools.rs:370-420`
  - `handle_create_bean` bypasses the fail-first check, hook execution, and verify suggestion that `cmd_create` provides. MCP-created beans never get `fail_first: true`, never trigger pre/post-create hooks, and never calculate tokens the same way. This is a behavioral divergence that could cause subtle bugs when mixing CLI and MCP usage.
  - Suggested fix: Share creation logic via a common function in the API layer. At minimum, MCP create should call the same `cmd_create` logic or a shared inner function.
- [high] test-coverage — `src/mcp/tools.rs:1-846`
  - Zero tests for 846 lines of MCP tool handlers. These handlers implement critical operations (create, close, claim, verify) that have behavioral divergence from the CLI commands they shadow. Bugs in MCP tools would only be discovered by manual testing with an IDE client.
  - Suggested fix: Add integration tests that exercise each MCP tool handler with mock beans directories, similar to the existing `commands/close.rs` test suite. Focus especially on `handle_close_bean` and `handle_create_bean` since they reimplement core logic.
- [high] edge-cases — `src/mcp/tools.rs:843-848`
  - `truncate_str` slices a `&str` at byte offset `s.len() - max` without checking for UTF-8 character boundaries. If the offset falls inside a multi-byte character (e.g., emoji, CJK, or the U+FFFD replacement character from `from_utf8_lossy`), this will panic at runtime with a "byte index is not a char boundary" error.
  - Suggested fix: Use the same `is_char_boundary` approach already implemented in `truncate_to_char_boundary` in close.rs. Replace with: `let start = s.len() - max; let start = (start..s.len()).find(|&i| s.is_char_boundary(i)).unwrap_or(s.len()); format!("...{}", &s[start..])`
- [high] edge-cases — `src/mcp/tools.rs:491-493`
  - In `handle_close_bean`, the output truncation `&combined[combined.len() - 2000..]` slices at a raw byte offset. If the 2000th-from-end byte lands inside a multi-byte UTF-8 character, this panics. The `combined` string comes from `from_utf8_lossy` which may insert 3-byte U+FFFD replacement characters.
  - Suggested fix: Add a char-boundary search: `let start = combined.len() - 2000; let start = (start..combined.len()).find(|&i| combined.is_char_boundary(i)).unwrap_or(combined.len()); format!("...{}", &combined[start..])`. Or extract a shared `truncate_tail(s, max)` helper alongside the existing `truncate_str`.
- [medium] architecture — `src/mcp/tools.rs:730-755`
  - `all_children_closed` and `auto_close_parent` are duplicated between `commands/close.rs` and `mcp/tools.rs` with different implementations. The `close.rs` version rebuilds the index fresh and includes archived beans; the `mcp/tools.rs` version only checks the current index and doesn't include archived beans. This means auto-close-parent can behave differently depending on whether you close via CLI or MCP.
  - Suggested fix: Consolidate into a single shared implementation. The `close.rs` version is more correct (includes archived beans).
- [medium] input-sanitization — `src/mcp/tools.rs:487-488`
  - The MCP close handler truncates combined output with `&combined[combined.len() - 2000..]` using byte indexing. If the verify command produces multi-byte UTF-8 output, slicing at byte 2000 from the end may split a multi-byte character, causing a panic. The MCP server would crash, disrupting IDE integration.
  - Suggested fix: Use `combined.char_indices().rev().nth(2000)` or similar to find a valid character boundary for truncation.
- [low] input-validation — `src/mcp/tools.rs:354-418`
  - The MCP `create_bean` handler accepts a `deps` parameter as a comma-separated string and splits it into dependency IDs, but does not validate each individual dependency ID with `validate_bean_id()`. A malicious MCP client could inject path-traversal characters in dependency IDs that would be stored in the bean file and later used in file lookups.
  - Suggested fix: Call `crate::util::validate_bean_id()` on each dependency ID parsed from the `deps` parameter before storing them in `bean.dependencies`.

- Prefer small, focused changes.
