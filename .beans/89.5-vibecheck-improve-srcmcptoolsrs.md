## Task
Improve `src/mcp/tools.rs` based on vibecheck findings.

## Findings (top 20)
- [high] input-validation — `src/mcp/tools.rs:377-382`
  - `truncate_str` slices a string at an arbitrary byte offset without checking UTF-8 char boundaries. The line `&s[s.len() - max..]` will panic at runtime if the slice point falls inside a multi-byte UTF-8 character (e.g., emoji or non-ASCII text in verify command output). The close.rs module has a correct `truncate_to_char_boundary` helper, but the MCP tools module doesn't use it.
  - Suggested fix: Replace the raw byte-offset slice with a char-boundary-safe truncation. Either reuse `truncate_to_char_boundary` from close.rs (move it to util.rs), or use `s.floor_char_boundary(s.len() - max)` (nightly), or walk backward to find a safe boundary like close.rs does.
- [low] error-handling — `src/mcp/tools.rs:218-235`
  - The MCP `close_bean` handler runs verify via `sh -c` but doesn't apply the same trust checks, hook execution, or structured history recording that the CLI `cmd_close` does. This creates a divergent code path where closing via MCP skips pre-close hooks, doesn't record `RunRecord` history, and doesn't fire post-close hooks or process `on_close` actions. An agent using MCP could bypass security hooks that the CLI enforces.
  - Suggested fix: Refactor the close logic into a shared function that both CLI and MCP handlers call, ensuring hooks, history recording, and on_close actions are consistently applied regardless of entry point.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
