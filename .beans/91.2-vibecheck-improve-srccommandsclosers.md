## Task
Improve `src/commands/close.rs` based on vibecheck findings.

## Findings (top 20)
- [high] single-responsibility — `src/commands/close.rs:267-680`
  - `cmd_close` is a ~410-line function that handles at least 8 distinct responsibilities: ID validation, bean loading, pre-close hook execution, verify command execution, failure recording with circuit breakers, on_fail action processing, worktree merge handling, archival, post-close hook execution, on_close actions, worktree cleanup, parent auto-close, and index rebuild. This makes it very difficult to reason about, test individual paths, or modify safely.
  - Suggested fix: Extract the verify-and-record logic into a dedicated function (e.g., `run_verify_and_record`), the archive logic into `archive_bean`, and the post-close actions into `run_post_close_actions`. The main `cmd_close` should orchestrate these phases as a pipeline: load → pre-hook → verify → close → archive → post-hook → cleanup → auto-close parent.
- [medium] edge-cases — `src/commands/close.rs:159-232`
  - `auto_close_parent` uses unbounded recursion to walk up the parent chain (calls itself recursively via grandparent check). A deeply nested bean hierarchy (e.g., 1000 levels deep) would stack overflow. While unlikely in practice, the `find_root_parent` function (line 236) already handles the same traversal iteratively — the inconsistency suggests this wasn't considered.
  - Suggested fix: Convert `auto_close_parent` to iterative traversal, similar to `find_root_parent`. Collect parent IDs to auto-close in a loop, then close them sequentially.
- [low] performance — `src/commands/close.rs:121-150`
  - `all_children_closed` rebuilds the entire index from disk (`Index::build(beans_dir)`) plus collects all archived beans (`Index::collect_archived`) on every call. This is called once per bean being closed AND once per parent in the auto-close chain. Closing 10 sibling beans would rebuild the index 20+ times (once per close + once per parent check).
  - Suggested fix: Accept an `&Index` parameter or rebuild the index once in `cmd_close` and pass it through. The index is already rebuilt at the end of `cmd_close` — the intermediate rebuilds are wasteful.
- [low] consistency — `src/commands/close.rs:159-232`
  - `auto_close_parent` duplicates the archive logic from `cmd_close` (compute slug, determine extension, build archive path, create directories, rename file, update is_archived, write to archive). Any change to the archival process must be updated in two places.
  - Suggested fix: Extract a shared `archive_bean(beans_dir, bean, bean_path)` function that both `cmd_close` and `auto_close_parent` call. This also makes the archival logic independently testable.
- [low] resource-cleanup — `src/commands/close.rs:54-74`
  - `run_verify` captures the full stdout and stderr of verify commands into memory via `.output()`. While `MAX_OUTPUT_BYTES` is defined (64 KB), it's only used for the `outputs` field truncation, not for limiting the actual memory consumption during capture. A verify command that produces gigabytes of output would cause OOM before any truncation happens.
  - Suggested fix: Replace `Command::output()` with a spawned process with piped stdout/stderr and read with a bounded buffer. Alternatively, document this as a known limitation for verify command authors.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
