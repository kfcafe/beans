## Findings (top 20)
- [high] single-responsibility â€” `src/commands/run.rs:1-1465`
  - `run.rs` is 2216 lines containing 19 functions spanning at least 4 major concerns: dispatch planning (`plan_dispatch`, `compute_waves`), template-mode execution (`run_wave_template`), direct-mode execution (`run_wave_direct`, `run_single_direct`, `run_ready_queue_direct`), and output formatting (`print_plan`, `print_plan_json`, `format_duration`). The ready-queue scheduler alone is ~170 lines of concurrent logic with channels and thread management.
  - Suggested fix: Split into submodules: `run/plan.rs` (dispatch planning, wave computation), `run/direct.rs` (pi spawning and monitoring), `run/template.rs` (legacy shell template spawning), `run/format.rs` (output helpers). Keep `run/mod.rs` as the thin orchestrator containing only `cmd_run`, `run_once`, and `run_loop`.
- [medium] architecture â€” `src/commands/run.rs:1463-1466`
  - `run.rs` re-exports `find_bean_file` as `pub fn find_bean_file(beans_dir, id)` that just calls `crate::discovery::find_bean_file`. This creates a confusing import path where the same function is available from two modules. The comment says "Public wrapper for use in tests" but it's `pub`, not `pub(crate)` or test-only.
  - Suggested fix: Remove the wrapper. Tests should import directly from `crate::discovery::find_bean_file`. If test-only access is needed, use `#[cfg(test)]` or `pub(crate)`.
- [medium] resource-cleanup â€” `src/commands/run.rs:997-1010`
  - In `run_ready_queue_direct`, spawned threads are not joined on early exit (the `!keep_going` failure path). After draining the channel, the function returns but orphaned threads may still be running `run_single_direct`, which spawns `pi` child processes. Those threads hold references to `beans_dir` PathBuf clones and may continue writing to the filesystem after the parent function returns.
  - Suggested fix: Track thread `JoinHandle`s and join them before returning, even on the failure path. The current `wait for currently running agents to finish` comment only drains the channel â€” it doesn't join threads, so there's a race where a thread could still be spawning a process when the function returns.
- [low] consistency â€” `src/commands/run.rs:816-888`
  - `run_wave_direct` and `run_ready_queue_direct` implement two different parallelism patterns for the same spawn mode (Direct). `run_wave_direct` uses `Arc<Mutex<Vec>>` for collecting results while `run_ready_queue_direct` uses `mpsc::channel`. The wave-based function polls `is_finished()` while the queue-based function uses blocking `recv()`. These inconsistent patterns make the code harder to maintain.
  - Suggested fix: Standardize on the channel-based approach (used by `run_ready_queue_direct`) since it's the more robust pattern. `run_wave_direct` is only called from the legacy `run_wave` dispatcher â€” consider whether it's still needed given the ready-queue mode.
